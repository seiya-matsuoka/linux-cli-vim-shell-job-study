# Phase 6 Performance Observation

## 目的

- 総合テストや性能確認時に、基本的な観測コマンドを使って状況を把握できるようにする
- CPU、メモリ、ディスク、ネットワークの観測を一通り経験する
- 軽い負荷時に観測結果がどう変わるかを説明できるようにする

## 学習項目

- `top`
- `htop`
- `uptime`
- `vmstat`
- `free`
- `df`
- `du`
- `ss`
- `curl`
- `iostat`
- `pidstat`
- 必要に応じた軽い負荷観測

## 使用環境・ツール

- VirtualBox
- Ubuntu Server VM
- htop
- sysstat
- curl
- stress-ng

## 学んだこと

### `top`

`top` は、システム全体の要約とプロセス一覧をリアルタイムで確認する入口として使いやすい。  
CPU、メモリ、load average をざっと見るのに向いている。

### `htop`

`htop` は `top` より見やすく、プロセス一覧や CPU / メモリ状況を対話的に追いやすい。  
学習用としては、まず `top` と `htop` の両方を見て、どちらも最低限触れることが重要。

### `uptime`

`uptime` はロードアベレージ確認の基本。  
短い出力で、稼働時間と 1 / 5 / 15 分平均の負荷が分かるので、負荷時の入口確認に便利。

### `vmstat`

`vmstat` は CPU だけでなく、プロセス数、メモリ、ページング、I/O もまとめて見られるので、総合的な状態把握に向いている。  
baseline と負荷時を比べると変化が分かりやすい。

### `free`

`free -h` はメモリ確認の基本。  
特にメモリ負荷をかけた前後比較が分かりやすい。

### `df`

`df -h` はファイルシステム単位の空き容量確認に使った。  
「どのマウントポイントがどれくらい埋まっているか」を見る基本として理解した。

### `du`

`du -sh` はディレクトリ単位の使用量確認に使った。  
`df` と `du` は役割が違い、`df` はファイルシステム、`du` はディレクトリという整理をした。

### `ss`

`ss -tlnp` は待ち受けポート確認に使った。  
ローカル HTTP サーバーを起動して、8080 番が listen していることを確認した。

### `curl`

`curl` はローカル HTTP サーバーへの応答確認に使った。  
ネットワーク疎通確認だけでなく、アプリが HTTP を返しているかの確認にも使えると理解した。

### `iostat`

`iostat -xz` は I/O 観測の基本として使った。  
ディスク I/O を観測するには、負荷をかける前後で比較するのが分かりやすい。

### `pidstat`

`pidstat` は特定プロセスに注目したいときに便利。  
HTTP サーバーや stress-ng の PID を指定して観測すると、「どのプロセスがどれくらい使っているか」を見やすい。

### 軽い負荷観測

今回の学習では、以下の軽い負荷をかけた。

- CPU 負荷
- メモリ負荷
- I/O 負荷

それぞれに対して、観測コマンドの出力がどう変わるかを確認した。

## 実際にやったこと

- baseline で `top`, `htop`, `uptime`, `vmstat`, `free` を確認した
- `df`, `du` でディスク使用量を確認した
- ローカル HTTP サーバーを立て、`ss` と `curl` を確認した
- `iostat`, `pidstat` で I/O / プロセス統計を確認した
- `stress-ng` と `dd` で軽い負荷をかけ、観測結果の変化を見た

## メモ

- まず baseline を観測する
- その後に軽い負荷をかけて差分を見る
- CPU、メモリ、ディスク、ネットワークで見るコマンドを分けて考える
- `df` と `du` は役割が違う
- `ss` と `curl` を組み合わせると、待ち受けと応答確認をセットで見られる
- `pidstat` はプロセス単位の確認に便利
